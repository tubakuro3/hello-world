# ソフトウェア工学第1回~6回まとめ

## ソフトウェアとは
　ソフトウェアとは、情報を扱うプロダクトそのものであり、プロダクトを提供する手段である。ソフトウェアの特徴として、以下の要素が挙げられる。まず、新しい環境や技術のニーズを満たすように適応しなければならない。また、新しいビジネス要求を実現するために強化されなければならず、より近代的なシステムやデータベースと相互運用するため拡張されなければならない。


### ソフトウェア工学の必要性
　ソフトウェア工学は、**品質**、**コスト**、**納期**の最適なバランスを実現するための手法・方法論である。ソフトウェア工学を学ぶべき理由として以下の4つが挙げられる。
1. 大規模化と複雑さ
2. IT人材の不足
3. 社会性と求められるミッション
4. 不確実性の増大


1について、現代社会はソフトウェアなしでは成り立たない。よって、ソフトウェアを効率よく開発し、メンテナンスする必要がある。そして、効率と品質の両立のためにはソフトウェア工学が必要である。2については、量的問題と質的問題がある。ソフトウェアの需要に対してIT人材が追い付ておらず、また、効率的に開発・管理することのできる人材も少ないという問題がある。3は、ライフライン・インフラストラクチャーとしてのソフトウェアが増加によて、不測の事態に対応した事業・サービスの継続が求められる。4は、ソフトウェア利用の変化・多様化によって、検索型から交流型への変化、意図しないソフトウェア活用が増えたことにより、ニーズに対応したソフトウェアの開発が必要になっている。これらの要因によって、ソフトウェア工学が必要である。


## ソフトウェアライフサイクル
ソフトウェアライフサイクルは以下の流れがある。
- ソフトウェアの誕生
- ソフトウェアの開発・運用
- ソフトウェアの廃止


まず、ニーズが発生し、ニーズを満たすために、ビジネス上あるいはシステム上の要求を具現化するための企画・計画を行う。そして、要件定義をまとめる。要件定義では、どんなソフトウェアが必要か、そのソフトウェアで業務がどのように変わるかについて決める必要がある。そして、ソフトウェアの開発を進め、実稼働する。その後、保守運用までする必要がある。サービスが終了したら、ソフトウェアの廃止であり、新規ソフトウェアへリニューアルする。

### 要件定義
要件定義とは、実現する機能・実現しない機能を明確にすることである。実現性評価としてFeasibility Studyがある。機能の絞り込み、スケジュールの見直し、代替案の検討などにつなげていく。企業によっては要件定義項目をテンプレート化し、もれなく検討ができる工夫をしている企業もある。


**要件定義の意義**
- プロジェクト関係者のバイブル
- 社内外への宣言と協力依頼
-　Request for Proposal(RFP)

### 制作
設計では、設計書に従ってシステム構築をする。内製では、一部機能を機能パッケージとして購入したり、外部委託することもある。外部委託は、開発を外部に委託することである。ソースコードを納品物に含めるか否か分かれる。含める場合、メンテナンスは自社で可能だが、含めない場合は、メンテナンスは作業委託契約を締結する必要がある。

### テストとデバッグ
バグのないソフトウェアはない。そのため、あらかじめテストしバグを早期に発見・対処することが理想である。すべてをテストすることは、工数に限界があるため、不可能である。したがって、ソフトウェアのどの箇所をテストするか、どの程度テストするか考える必要ある。


## プロジェクト
プロジェクトは、必ず終わりがある類の仕事である有期性と独自の木庭を達成する類の仕事である独自性がある。

### フォアキャスティング
フォアキャスティングは、今の立ち位置から順に目的に進んでいく。しかし、目標に到達できない可能や、目標と遠いところに到達する可能性がある。

### バックキャスティング
バックキャスティングは、定めた目標にむけて何が必要がを計画し達成に向かうことである。目標を明確に設定する必要があり、目標に向けての筋道を検討する必要がある。

## ソフトウェア分析
### ソフトウェアをみえる化するための手法
- ソフトウェアの物量に関する分析
- ソフトウェアの実行性能に関する分析
- ソフトウェアの開発工数に関する分析
- ソフトウェアの品質に関する分析

### ソフトウェアの評価
ソフトウェアの評価は、価値を測ることである。評価は以下の観点から測ることができる。

**1.コードの物量(ステップ数)**

換算でのソースコード行数。複数言語が入り混じっている場合は菅さんする必要がある。

**2.コードの物量** 

組み込み型ソフトウェアはサイズが小さいほど良い。

**3.ファンクションポイント法**

プログラミングフェーズに入る前にユーザ要件が定まり、必要な機能が見えてきた段階でシステム規模を換算することができる。
- 大まかな見積もり>使用の設計>再見積もり>システム規模の見積もり
- システム調達の際の異なるステムの規模を比較、新規開発するか購入するかなどの投資意思決定の指標にも利用できる。

## 開発プロセス
### ウォーターフォール型
上流から下流まで順に開発を進める手法である。進捗管理が容易であり、成果物が明確であるが、後工程にしわ寄せが集中してしまうリスクがある。

### スパイラルモデル
プログラム開発を小さなフェーズに分割する。フェーズごとにプロトタイプによるデモンストレーションをし、フィードバックを重ねる。プロトタイプ作成に想定外の作業量が発生するリスクがある。

### アジャイルプロセス
機能ごとに小さな単位で、計画・設計・実装・テストを繰り返す手法である。変化に応じて柔軟に対応することができる。モチベーションをあげ、学習効果を高めることで無駄をなくす。

## WBS
WBSとは、Work Breakdown Structureの略であり、プロジェクト目標を達成し、必要な要素成果物を生成するために、プロジェクトチームが実行する作業を、要素成果物を主体に階層的に要素分解したものである。プロジェクトを細かな作業に分解した構成図。作業を細かく分解し、作業順に並べる。メリットとしてスコープが明確になり、作業を洗い出すことができる。また、全体管理と作業計画が明確化される。
